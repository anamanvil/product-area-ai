2 Issues Introduced by v1 → v2 Migration

Breaking responseId loses meaning

In v1 (Responses API), each response had a unique id from OpenAI, enabling previous_response_id chaining. In v2, agent-runner.ts sets responseId: conversationId as a placeholder for every event. This means AIAgentToolInvoked, AIAgentToolExecuted, and AIAgentResponse all share the same responseId across the entire conversation.

Impact: Any dashboard or query that groups by responseId to count "per-turn" metrics will collapse all turns into one bucket.

Recommendation: Generate a turnId = crypto.randomUUID() per request in the v2 middleware. Pass it to runAgent and use it in place of responseId. This restores per-turn correlation.

Degraded previousResponseId permanently undefined

The v2 path never sets previousResponseId. The event factory still accepts it, but it will always be empty. During the v1/v2 coexistence period, analytics queries that rely on this field to reconstruct conversation chains will produce incomplete data for v2 traffic.

Recommendation: Acceptable to drop since conversationId + turnId provide equivalent correlation. Document the deprecation for analytics consumers.

New isNewConversation added but not fully wired

The PR adds isNewConversation?: boolean to the AIAgentRequest event payload type. This is useful data (first turn vs follow-up) that was previously inferred from previousResponseId === undefined. Verify the event factory actually persists this to PubSub and that the analytics schema can receive it.

Recommendation: Confirm the buildAIAgentRequestEvent factory in commons propagates this field. If not, add it.

Inconsistency Action format change not reflected in events

The v2 stream contract changes action references from {id, value} to {callId, result} (per sc-541427). The EventsService type still uses Action = { id, name, args }. The v2 runner maps callId → id internally, which works, but the frontend AIAgentToolExecuted event may need updating to match.

Recommendation: Audit the frontend AIAgentToolExecuted event in useGenAiActions.ts after sc-541427 merges to confirm the tool identifier field name is consistent.

3 Structural Problem: Everything Is AIAgent*

With the Agent Config Assistant now shipping and more AI features on the roadmap, all features share the same AIAgent* event names. There is no way to distinguish which feature generated an event in analytics.

Builder Agent

/feature/aiagent/aiagent
  • Main AI agent for map interaction
  • Has mapId, tools, delegated actions
  • Uses genAiHooks.ts → useChatAgent
  • Fires all AIAgent* events

Agent Config Assistant New

/feature/aiagent_assistant/aiagent_assistant
  • Helps users configure AI agent prompts & tools
  • Has mapId, different tool set
  • ai_feature: aiagent_assistant in prompt config
  • Uses useBuilderAssistantChatAgent → useChatAgent
  • Fires same AIAgent* events — indistinguishable from Builder Agent

Future AI Features

/feature/{x}/{y}
  • SQL Assistant, Data Explorer AI, etc.
  • Will likely use useChatAgent
  • Will fire same AIAgent* events
  • No way to distinguish in telemetry

The only current way to differentiate is by mapId presence or by cross-referencing the feature path from HTTP logs—neither of which is in the event payload itself.

4 Proposed Event Consistency System

Design Principles

Backend events = execution telemetry (what the system did) Frontend events = interaction telemetry (what the user did) Common fields = shared context (ties them together)
A Add feature as a first-class field

Every backend and frontend AI event should include a feature discriminator so analytics can filter and compare across AI products. 

feature: 'aiagent' | 'aiagent_assistant' | 'sql-assistant' | string
Key insight: ai_feature already exists in prompt config. Each prompt file declares its ai_feature value (e.g. aiagent, aiagent_assistant). This already flows to AIConsumedEvent and AIRequestGotOverquotaEvent via the metrics/quota pipeline. The proposal is to use the same ai_feature value as the feature discriminator on all AI events (Request, Response, ToolExecuted, etc.), so every event in BQ is filterable by feature.

Backend: The feature name is already in the URL path (/v2/feature/:feature) and in the prompt config (ai_feature). Extract it in the middleware and pass through to EventsService.send().

Frontend: useChatAgent already receives agentId (e.g., 'aiagent/aiagent'). The wrapper hook maps this to the canonical ai_feature value and passes it to sendMetricEvent.

Complexity: — Small plumbing change across both layers.

B Replace responseId with turnId in v2

Generate a turnId per incoming request to restore per-turn correlation that responseId provided in v1. 

// In agent-middleware-agents-sdk.ts
const turnId = crypto.randomUUID()

// Passed to runAgent, used in all EventsService.send() calls
responseId: turnId  // reuse existing field name for backward compat

Complexity: — Single change point in middleware.

C Rename frontend events to feature-generic AI*

Chat-related events should drop the "Agent" prefix since they apply to any AI chat feature. The feature field provides the discriminator.

Migration: Ship new events in parallel, deprecate old ones after dashboards are updated.

Complexity: — Touches many components but each change is small.

D Move common event emission into useChatAgent

The useChatAgent hook is the single entry point for all AI chat features. Common chat events should be fired here, so every feature gets tracking for free.

useChatAgent (fires: ConversationStarted, ResponseStopped, ToolExecuted) | |-- useBuilderChatAgent (fires: agent-specific events only) |-- useAssistantChatAgent (fires: assistant-specific events only) `-- useFutureChatAgent (gets common tracking for free)

Currently genAiHooks.ts fires events like AIAgentConversationStarted that should live in useChatAgent. Moving them down ensures new features don't need to re-implement tracking.

Complexity: — Refactor with testing, after sc-541427 merges.

E Flat, typed event schema — every field named and documented

Not all fields apply to all features. mapId is for map-bound features (Builder Agent, Agent Config Assistant), mode (testing/live) is agent-only, workflowId is for future workflow AI, etc. Every field must be explicitly named and typed so that anyone querying events in BigQuery or Segment knows exactly what columns exist and what they mean. The feature value should match the ai_feature declared in each prompt config file.

Single flat type — all fields explicit, feature-specific ones optional

type AIEventPayload = {
  // ── Always present on every AI event ──────────────────────
  feature: string           // 'aiagent' | 'aiagent_assistant' | ... (matches ai_feature in prompt config)
  conversationId: string    // session-level correlation
  turnId: string            // request-level correlation (replaces responseId)
  accountId?: string        // tenant

  // ── Resource context (null when not applicable) ──────────
  mapId?: string            // which map — aiagent, aiagent_assistant
  agentId?: string          // which agent config — aiagent only
  workflowId?: string       // which workflow — workflow-ai
  connectionId?: string     // which connection — sql-assistant

  // ── Execution context ────────────────────────────────────
  model?: string            // which LLM was used
  mode?: 'testing' | 'live' // aiagent only (has test/live toggle)
  isNewConversation?: boolean // first turn in conversation
}
No opaque bags. Every field is explicitly named and typed. This is the schema that appears in BigQuery and Segment. People querying events can SELECT mapId, mode, feature directly — no JSON parsing, no guessing.

Example BQ queries that Just Work

-- "How many agent conversations per map, in live mode only?"
SELECT mapId, COUNT(DISTINCT conversationId) AS conversations
FROM ai_events
WHERE feature = 'aiagent' AND mode = 'live'
GROUP BY mapId

-- "Tool success rate per feature"
SELECT feature, toolName,
  COUNTIF(success) / COUNT(*) AS success_rate
FROM ai_events
WHERE event_type = 'AIAgentToolExecuted'
GROUP BY feature, toolName

-- "All AI activity on a specific map (agent + assistant)"
SELECT feature, event_type, turnId, timestamp
FROM ai_events
WHERE mapId = 'map-abc'
ORDER BY timestamp

How it flows in the backend:

v2 middleware receives request → extracts feature from URL path (/v2/feature/:feature) → generates turnId = crypto.randomUUID() → populates named fields from request: mapId, agentId, mode, etc. → fields not applicable to this feature are simply omitted (null in BQ) → constructs AIEventPayload and passes to runAgent() → EventsService.send() includes the payload in every event
